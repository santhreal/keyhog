System triage

When an incident hits and you need to know "are there leaked credentials on this host, anywhere?", keyhog scan-system walks every mounted drive and every git history under a hard space ceiling.

Basic usage

$ sudo keyhog scan-system --space 50G

Enumerates every mounted drive (skipping pseudo-FS like /proc, /sys, tmpfs, nsfs, fuse.snapfuse), auto-discovers every .git directory (worktrees, bare repos, submodules), and runs the same scan + git-history pipeline that keyhog scan --git-history uses on each. Honors a hard --space <bytes> ceiling.

Space ceiling

The --space flag is the hard stop. keyhog tracks bytes-read in real time and halts the scan cleanly when the ceiling is hit. Suffixes: K / M / G / T.

$ keyhog scan-system --space 1G        # quick smell test
$ keyhog scan-system --space 50G       # default — most laptops finish
$ keyhog scan-system --space 1T        # thorough; servers

Findings up to the ceiling are emitted. The scan ends with a summary of how much was actually scanned vs how much was skipped due to the ceiling, so you know whether a follow-up run is warranted.

Network mounts

$ keyhog scan-system --space 1T --include-network

By default keyhog skips NFS / SMB mounts to avoid pulling gigabytes off the network and producing host-wide unexpected load. Add --include-network for true machine-wide coverage; pair with --space to keep the bandwidth bounded.

Git history

$ keyhog scan-system --space 10G --no-git-history

Scans only working trees; skips the historical blob walk. Useful for a fast first-pass — git history can dominate runtime on long-lived repos.

Output for triage

$ keyhog scan-system --space 50G --output findings.json --format json
$ jq '.findings | group_by(.detector) | map({detector: .[0].detector, n: length})' findings.json

The JSON shape is identical to a normal scan (see Output formats). Extra metadata at the top level reports mount enumeration, skipped paths, and the space-ceiling hit status.

What gets ignored

System triage is paranoid and bounded:

  • Ignores .gitignore by default. An attacker stashing a leaked key would .gitignore it; the triage scan refuses to be fooled. Add --respect-gitignore if you want normal-scan behavior on developer laptops.
  • Skips pseudo-filesystems. /proc, /sys, tmpfs, cgroup, nsfs, fuse.snapfuse, devfs, autofs. The triage scan is for persistent storage.
  • Skips files above --max-file-size (default 10 MB). Override for video / disk-image scanning.
  • Skips known-binary types (images, archives, compiled binaries) unless --scan-binary is passed.

Always-on hardening applies

System scans always run with the always-on hardening flags (no core dumps, no ptrace, PR_SET_DUMPABLE=0). The scan can spend hours touching every file on the box; the hardening makes sure a hostile debugger can't snapshot the in-memory findings.

Add --lockdown for the full sealed-host treatment (mlockall, refuses to start with a stale cache, refuses --show-secrets). See Lockdown mode →.

Permissions

Without root, scan-system reads what your user can read — which on a multi-user system means you'll miss other users' home directories. Run under sudo for true machine-wide coverage:

$ sudo keyhog scan-system --space 50G --output /tmp/triage.json

keyhog never writes anywhere except the output path you specify and the on-disk cache under ~/.cache/keyhog/ (which --lockdown refuses to populate). Inspect the cache layout with keyhog cache info.

Use cases

Incident response

"A laptop was lost. What credentials were on it?" Run on a fresh image of the disk; the JSON output is your evidence trail.

Build-host audit

"Did our CI runner ever write secrets to disk?" Scan /var/lib/jenkins, /var/lib/buildkite-agent, etc.

Developer hygiene

"Are there any leaked customer credentials on this dev laptop?" Quarterly self-scan; commit a triage report to a private repo.

M&A inheritance

"We just acquired a codebase — what's leaked across all their history?" scan-system on a backup snapshot, results shipped to the security team.
