FAQ

The questions teams ask before adopting keyhog. If yours isn't here, file an issue — answers land in the docs.

Does keyhog send anything over the network?

No. Default operation is fully offline. There is no telemetry, no usage reporting, no "phone home." The only network calls keyhog ever makes are:

  • --verify — explicit live verification. Disabled by default; refused under --lockdown.
  • --github-org / --s3 sources — explicit data sources where you've configured what to fetch.
  • Release update check — disabled by default, opt in with keyhog config set updates.check true.

Run with strace -e network if you want to verify the wire silence.

Are credentials sent anywhere?

No. Even with --verify, only the credentials you've explicitly asked to live-verify are sent — read-only, to the matching vendor API only, with the vendor's own auth (which is the credential being verified). Verifiers are pinned to specific endpoints; no wildcard fallback.

How many false positives should I expect?

On the Django comparison corpus (86 MB, 0 real secrets), keyhog fires 1 false positive. Most teams adopting keyhog see a one-time wave of findings at adoption (often already-rotated leaks in git history), which they resolve with a baseline, and then ~5 false positives per 100k LOC scanned thereafter. Calibrate with keyhog calibrate to drop per-detector confidence in your codebase further.

What about secrets I want to allowlist (test fixtures, examples)?

Drop the path, hash, or detector id in .keyhogignore — see Allowlists →. Add reason + expires for governance.

Why Rust?

Three reasons keyhog is in Rust and not Go / Python / Node:

  • Single static binary, no runtime to install, ships on every platform with the same behavior.
  • Hyperscan / vyre / wgpu bindings are first-class in the Rust ecosystem; multi-pattern NFA at SIMD throughput is the killer primitive for this problem.
  • Memory safety + zero-cost abstractions make the --lockdown guarantees enforceable, not aspirational.

Why is the binary so big?

~28 MB stripped, ~110 MB unstripped Docker image. The detector database alone compiles to ~6 MB; Hyperscan + Vectorscan + wgpu account for most of the rest. cargo install keyhog --no-default-features --features regex-engine shrinks to ~3 MB at the cost of throughput and decode-through scanning.

Does keyhog have a SaaS / hosted version?

No. keyhog is a CLI + library you run on your own infrastructure. GitHub code scanning + SARIF gives you most of the "hosted dashboard" experience for free; pair with EnvSeal for the runtime side.

Is keyhog suitable for EU-CRA / SOC 2 / FedRAMP audits?

keyhog itself doesn't make you compliant — but it produces SARIF with CWE-798 + OWASP A07:2021 taxa, which most auditors accept as evidence of "we have automated secrets scanning in CI." The --lockdown + always-on hardening (no core dumps, no ptrace) covers the "scanner must not be a credential exfil vector" requirement that some auditors ask about. Pair with a baseline + retention policy for the SARIF outputs.

Can I run keyhog on Windows?

Yes — Linux/macOS/Windows all supported. On Windows the SIMD path is opt-in today (--features simd) because vcpkg/MSVC build of Hyperscan is required; the AVX2 regex fallback works out of the box and is ~5–10× slower per pattern. Native Windows Hyperscan via vendored build profile is on the roadmap.

Does keyhog support custom detectors written in Rust?

Not as a stable extension API today. The supported way to add detectors is TOML — see Detector catalog →. For complex logic that doesn't fit TOML's regex + companion model, filing an issue with the use case is the path; many "needs custom Rust" asks turn out to be expressible as TOML once we look at them.

How do I report a security issue in keyhog itself?

Email security@santh.dev (PGP key on the org page). Please don't file public issues for security bugs in keyhog itself; we'll acknowledge within 48 hours and ship a patch or workaround within 7 days.

How do I report a missed secret or a false positive?

File a GitHub issue with the redacted credential shape (first 4 + last 4 chars), the detector that should have fired (or that fired wrongly), and ideally a minimal reproducer. Each report becomes a positive + adversarial test under tests/contracts/; the engine improves; the fix ships in the next patch release.

License?

MIT. Use it commercially, embed it, fork it, sell a hosted version of it. The detector TOMLs are also MIT — adding a new detector is a 5-line contribution with zero legal friction.

Versioning?

Pre-1.0 — minor releases (0.X) may have breaking flag changes; breaking changes are called out in the CHANGELOG under a clear "BREAKING" header. The finding JSON schema is versioned independently and follows strict semver — schema changes are explicit.

Where is the source for this docs site?

github.com/santhsecurity/keyhog/tree/main/site. Static HTML + a tiny Python build script. Open a PR if you spot a mistake; the round-trip is short.
