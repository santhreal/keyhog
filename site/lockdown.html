Lockdown mode

For environments where keyhog runs on the same machine that holds the secrets (e.g. paired with EnvSeal) and there is no trusted boundary between the scanner process and the credentials it's inspecting — pass --lockdown.

What --lockdown enforces

$ keyhog scan . --lockdown
  • mlockall(MCL_CURRENT | MCL_FUTURE) on Linux. Credentials never page to swap. Requires CAP_IPC_LOCK or matching RLIMIT_MEMLOCK; refuses to start otherwise.
  • prctl(PR_SET_DUMPABLE, 0) (always on, even outside lockdown). Disables core dumps, ptrace attachment, and /proc/<pid>/mem reads. macOS gets the equivalent PT_DENY_ATTACH.
  • Refuses to run if ~/.cache/keyhog/* exists. A stale on-disk cache would persist hashes / fragments of past findings; lockdown insists you clear the cache first.
  • Disables --incremental cache writes. The Merkle index hashes file content; under lockdown, nothing about the scan is allowed to persist to disk.
  • Refuses --verify. Verification sends credentials to outbound HTTPS endpoints — the entire point of lockdown is that those network round-trips don't happen.
  • Refuses --show-secrets. Plaintext credentials in stdout are exactly what lockdown is for the absence of.
  • Refuses to start if kernel coredump_filter would dump anonymous pages. On Linux, this is /proc/self/coredump_filter; lockdown verifies it excludes anonymous + huge pages before continuing.

What's always on (no flag needed)

Even without --lockdown, every keyhog binary at startup:

  • Sets PR_SET_DUMPABLE = 0 (Linux) or PT_DENY_ATTACH (macOS) — no core dump, no ptrace.
  • Sets process resource limits to refuse coredump writes (RLIMIT_CORE = 0).
  • Wipes credential strings from memory via zeroize on drop.
  • Refuses to read environment variables containing credentials into log output (env-var assignments get redacted before any logging).

Lockdown adds the sealed-host invariants on top of these always-on hardenings.

When to use it

EnvSeal pairing

keyhog runs inside a sealed environment alongside the credentials it inspects. No untrusted code on the box. --lockdown is the contract that says "and we'll keep it that way."

Incident-response triage

Scanning a forensic disk image, you don't want the scanner to write anything (cache, baseline, output) to the analysis host that could contaminate the evidence chain.

Multi-tenant CI runners

Other jobs share the runner. mlockall + no-cache prevents another job from reading swap-paged credentials or stale cache entries.

Air-gapped scanning

The scanner must not attempt outbound verification. Lockdown refuses --verify by configuration, so a misconfigured job can't leak credentials over a side channel.

Enforce lockdown in CI

Set the config or env to make lockdown mandatory:

[lockdown]
require = true           # in .keyhog.toml — refuse to run without --lockdown
$ KEYHOG_LOCKDOWN_REQUIRE=1 keyhog scan .
ERROR: lockdown is required by configuration but --lockdown was not passed
  rerun with: keyhog scan . --lockdown
  to disable enforcement: unset KEYHOG_LOCKDOWN_REQUIRE / remove [lockdown] require

Pair with --baseline + --format sarif so the sealed environment still produces actionable CI artifacts:

$ keyhog scan . --lockdown \
    --baseline .keyhog-baseline.json \
    --format sarif -o /dev/stdout \
    --severity high

What lockdown does NOT do

  • It does not encrypt the output. SARIF / JSON / text goes wherever you redirect it; if your CI artifact storage is unsealed, the output is unsealed.
  • It does not prevent kill -USR2 from dumping internal state if a debugger is attached before startup. PR_SET_DUMPABLE blocks attach post-startup; pre-startup is your container runtime's job.
  • It does not enforce a network boundary at the kernel level. If another process on the box exfiltrates findings via shared memory or the filesystem, lockdown can't see that. Treat lockdown as defense-in-depth, not the only defense.

Verifying it took effect

$ keyhog scan . --lockdown --verbose
lockdown: ENABLED
  mlockall: OK (locked 124 MiB)
  PR_SET_DUMPABLE: 0 (verified via /proc/self/status)
  coredump_filter: 0x33 (no anonymous, no huge pages) — OK
  cache: ~/.cache/keyhog/ absent — OK
  --verify: blocked
  --show-secrets: blocked
  --incremental: write-disabled (read still allowed for replay)
