Architecture

keyhog is a two-phase coalesced scanner with a hardware-routed backend. This page traces what happens between keyhog scan . and the first finding on your terminal.

Crate layout

crates/
  core/      Detector loading, findings types, reporting, allowlists
  scanner/   Hardware routing, Hyperscan, GPU, decode-through, entropy, ML, multiline
  sources/   File system, git (staged/diff/history), stdin, Docker, S3, GitHub org, web
  verifier/  Live credential verification against service APIs
  cli/       CLI binary, orchestration, baselines, benchmarks

Startup

  1. Hardware probe. cpufeatures for SIMD, wgpu::Instance::enumerate_adapters for GPUs. Result cached for the process lifetime.
  2. Detector load. 891 TOML files compiled in at build time via the build.rs embedding (auto-bundled in keyhog-core; no feature flag). Disk-resident detectors/ overrides the embedded set when present.
  3. Database compile. All regexes feed into hs_compile_multi; the resulting NFA serializes to ~/.cache/keyhog/db.bin with a SHA-256 over the detector set. Subsequent runs mmap the cached DB in <5 ms.
  4. Backend selection. The router picks GPU / SIMD / CPU per scan based on input size + pattern count (matrix in Performance →).

Two-phase coalesced scan

Naive multi-pattern regex blows up at 891 detectors. The trick is to split the work:

Phase 1 — broad-spectrum NFA over all files. Hyperscan fans the database across every chunk via rayon. The NFA is tuned for "is there any potential match here?" — it doesn't extract or score, it just flags hits. Empirically 95%+ of files have zero hits and pay zero post-process cost.

Phase 2 — full extraction on hits only. For each matched (file, detector) pair: regex capture groups, companion matching, checksum validation (GitHub CRC32, npm, Slack), entropy gating, ML confidence scoring. This phase runs only on the ~5% of files that survived phase 1.

Result: a 1 GB tree with a handful of leaks scans in ~2 s instead of the ~minutes a naive scanner needs.

Decode-through

Many leaks are encoded — base64 in a Kubernetes manifest, base64url in a JWT, hex in a Terraform variable file. keyhog peels these before regex matching:

  • Structured parsers. Kubernetes Secret YAML: parse, extract data:, base64-decode, append key: <decoded> lines so every downstream detector sees the plaintext. Same for Docker config.json (auth), Helm values, Vault KV templates.
  • JWT scanner. Split on ., base64url-decode each segment, re-scan. Catches JWTs with embedded service tokens (Stripe, Slack, GitHub) in the payload.
  • Generic base64 / hex scan. Any blob ≥ 20 chars with a recognizable charset gets a decode attempt; decoded bytes feed back through phase 2.

Multiline reassembly

Secrets split across lines (e.g. "sk-proj-" + \ continuation in JavaScript, or a literal newline inside a quoted YAML string) defeat naive line-by-line scanners. keyhog runs a lightweight join pass on:

  • Backslash-continuation lines (shell, Makefile, C).
  • String-concatenation in JS/TS ("a" + "b").
  • YAML/JSON multi-line strings (>-, |).
  • Templated Helm / Jinja that produces a literal token at render time.

Reassembled buffers feed phase 1 the same way raw chunks do.

Confidence scoring

Each finding gets a confidence in [0.0, 1.0] derived from:

  • Entropy. Shannon entropy of the credential bytes. Real secrets have higher entropy than placeholders.
  • Context. Is the surrounding text consistent with a real secret (env-var assignment, JSON object) vs an example (markdown code block, comment that says "fake")?
  • Companion match. AWS access key with a 40-char AWS_SECRET_ACCESS_KEY nearby is much higher confidence than the access key alone.
  • Checksum. GitHub PATs carry a CRC32; npm tokens carry a checksum; Slack tokens encode a workspace prefix. Invalid checksum → reject.
  • ML head. A small classifier (~30k params, trained on labeled true/false positives) re-scores the candidate against surrounding code context. Disabled with --fast.
  • Bayesian Beta(α,β) calibration. Per-detector posterior mean from keyhog calibrate feedback. Tells you "this detector is 0.8 of the time wrong in YOUR codebase" and damps accordingly.

Hardware backends

BackendHow it works
gpu-zero-copyvyre AC automaton on GPU cores. cudagrep NVMe → VRAM DMA bypasses the host CPU entirely for files > 256 MiB.
simd-regexHyperscan / Vectorscan NFA over AVX-512 / AVX2 / NEON. ~500 MB/s on a 16-core box.
cpu-fallbackAho-Corasick prefix filter + Rust regex for the survivors. Always available; correctness floor.
mega-scanSingle 256 MiB GPU shard for streaming large single files (HAR, log dumps).

Selection is automatic. The router probes once at startup, caches the result, and picks per scan based on input size and pattern count.

Memory + I/O profile

  • Memory: ~120 MB RSS on a typical scan (Hyperscan DB + scratch + ML weights). Daemon mode keeps this resident.
  • File I/O: mmap for files > 1 MB, buffered read otherwise. Direct I/O (O_DIRECT) for the GPU DMA path on Linux.
  • GPU VRAM: 256 MiB per shard; up to N shards in parallel where N is the device's max in-flight kernels.
  • Disk: compiled DB cache (~6 MB), incremental Merkle index (~1 KB per scanned file), nothing else.

Determinism

Same input → same output, byte-exact. Phase ordering, finding sort key, JSON field order are all deterministic. CI gates that compare scan outputs across commits will see only real diffs, never reordering noise.
