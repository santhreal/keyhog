Daemon mode

Every keyhog invocation pays a cold-start compile of 922 detectors into a single Hyperscan database. Run keyhog as a long-lived daemon and that cost is paid once per host; latency claims are reported by the daemon benchmark artifacts.

Start the daemon

$ keyhog daemon start
keyhog daemon started · pid 18244 · $XDG_RUNTIME_DIR/keyhog.sock

$ keyhog daemon status
running · uptime 14m22s · scans served 1,891 · current backend simd-regex

$ keyhog daemon stop

The socket defaults to $XDG_RUNTIME_DIR/keyhog.sock on Linux, /tmp/keyhog-$UID.sock on macOS. Override with --socket PATH.

Use it from any scan

$ keyhog scan --stdin --daemon=on < .env
$ keyhog scan ./one-file.env --daemon=on
$ keyhog scan --stdin --daemon=auto < .env

--daemon=on (and bare --daemon) is a hard contract: if the daemon is not running or cannot honor the exact scan, the command exits with an actionable error. Use --daemon=auto to opportunistically use a live daemon and otherwise run the in-process scanner, or --daemon=off to force the in-process path. The same binary handles both — there is no separate client.

Measured speedup

InputIn-processVia daemonSpeedup
Single GitHub PAT (32 bytes stdin)740 ms7 ms105×
1-file diff (typical pre-commit)1.1 s~12 ms91×
100-file diff1.8 s~85 ms21×
Full Django tree (86 MB)0.5 s0.4 s1.25×

The win is biggest on small inputs where the compile cost dominates. Once the input crosses ~50 MB the actual scanning time eclipses the saved cold-start and the daemon is barely faster than in-process. Pre-commit hooks and IDE save handlers are the killer use cases.

Where to run it

  • Developer machine: auto-start at login via a systemd user unit or launchd plist (see below). Hook into pre-commit / editor-on-save.
  • CI runner: start once at the beginning of the job script; tear down at the end. The same socket can serve hundreds of per-file scans for a large PR.
  • Long-running watchdog: pair with keyhog watch to get sub-100 ms re-scan on every file save under a directory.

systemd user unit

# ~/.config/systemd/user/keyhog-daemon.service
[Unit]
Description=keyhog secret-scan daemon
After=network.target

[Service]
Type=exec
ExecStart=/usr/local/bin/keyhog daemon start
Restart=on-failure
RestartSec=5

[Install]
WantedBy=default.target
$ systemctl --user daemon-reload
$ systemctl --user enable --now keyhog-daemon

launchd (macOS)

// ~/Library/LaunchAgents/dev.santh.keyhog.plist
<plist version="1.0">
<dict>
  <key>Label</key><string>dev.santh.keyhog</string>
  <key>ProgramArguments</key>
  <array>
    <string>/opt/homebrew/bin/keyhog</string>
    <string>daemon</string><string>start</string>
  </array>
  <key>KeepAlive</key><true/>
  <key>RunAtLoad</key><true/>
</dict>
</plist>
$ launchctl load ~/Library/LaunchAgents/dev.santh.keyhog.plist

Hot reload after a keyhog upgrade

$ curl -fsSL https://raw.githubusercontent.com/santhsecurity/keyhog/main/install.sh | sh
$ keyhog daemon stop && keyhog daemon start
keyhog daemon stopped
keyhog daemon started · loaded 922 detectors

KeyHog does not ship a single daemon restart verb. stop && start drops the old socket, recompiles the detector database under the new binary, and binds a fresh socket. Inflight scans on the old daemon complete normally before the old process exits.

Watch mode (editor on-save)

$ keyhog watch ./src
watching ./src · inotify/FSEvents/RDCW recursive
  src/auth.py modified → 0 findings (12 ms)
  src/config.ts modified → 1 finding [aws-access-key, line 4]

Connects to the daemon under the hood — every file save triggers a sub-100 ms re-scan. Use it for live IDE integration via a tail-watcher plugin, or just leave it running in a side terminal.
