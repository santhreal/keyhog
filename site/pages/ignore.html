Allowlists

Drop a .keyhogignore at your repo root to suppress individually known findings. Two syntaxes coexist — pick whichever is shorter for the case at hand.

Gitignore-style shorthand

The most common case: drop in a copied-over .gitignore and have it work. Bare globs are interpreted as path matches; bare 64-char hex hashes are interpreted as sha256 matches against the offending credential.

# comments start with #
*.log
node_modules/
vendor/**/*.json
.cache/

# bare 64-char hex → sha256 of the credential
9d6060e21ef8d5daec9cfe4a44b1b1bc9792246bfad28210edaaa1782a8a676a

Hashes are stable across keyhog versions: rotating to a new key produces a new hash, so allowlisting a rotated credential won't accidentally bless the next one. This is the same hash that appears in the SARIF partialFingerprints and the JSON sha256 field.

Explicit prefixes (governance)

For entries you want to track — who allowlisted it, when it expires, why it's safe — use the explicit-prefix form. The metadata is structured and machine-readable.

hash:9f86d081884c7d659a2feaa0c55ad015a3bf4f1b2b0b822cd15d6c15b0f00a08 \
  ; reason="rotated 2026-04-25, kept for git-blame trail" \
  ; expires=2026-07-01 \
  ; approved_by="security@acme.example"

detector:demo-token \
  ; reason="fixture detector — only used in docs/" \
  ; approved_by="docs-team"

path:**/fixtures/*.env \
  ; reason="snapshot test fixtures, all rotated" \
  ; expires=2026-12-31

Match rules

FormMatches when
*.logFinding's path matches the glob (gitignore semantics — ** = any depth)
node_modules/Finding's path is under the directory
<64-hex>sha256(credential) equals the bare hash
hash:<hex>Same, explicit form (supports metadata)
path:GLOBSame as bare glob, explicit form (supports metadata)
detector:IDDetector id equals ID (suppress an entire rule)

Expiration

Entries with expires in the past are silently dropped on load with a WARN line. This is the mechanism for "this is allowlisted for now, but force me to revisit it by Q3." If the allowlist auto-expires and the finding comes back, you can either re-allowlist with a new expiry or fix the underlying leak.

$ keyhog scan .
WARN: .keyhogignore:14: entry expired on 2026-04-01 — dropping
  hash:9d6060e2…782a8a676a  ; reason="…"  ; expires=2026-04-01

Approval audit

$ keyhog ignore audit
.keyhogignore   28 entries  ·  5 expire within 30 days  ·  0 expired
  by approver:
    security@acme.example     12
    docs-team                  6
    <unspecified>             10  ← consider requiring approved_by= for all entries

Where files are read from

keyhog walks up from each scanned file looking for .keyhogignore files, the same way git locates .gitignore. Nested files add to (don't override) parent files. The repo-root file is the highest-precedence override.

Override the file with --keyhogignore PATH. Disable allowlists entirely with --no-keyhogignore — useful when running a security audit that should ignore the team's own suppressions.

What does NOT belong here

  • Whole-repo "skip these directories" rules. Use .gitignore — keyhog respects it by default. Only put paths in .keyhogignore when you specifically want them scanned by every other tool but suppressed in keyhog.
  • "All findings from today's adoption." Use a baseline. Baselines track snapshots; the ignore file tracks individually-reviewed exemptions.
  • "All findings in test files." Use --exclude '**/test/**' at the CLI or add it to .keyhog.toml. The ignore file should be for individually- reviewed exceptions, not bulk filters.
