Baselines

Adopting keyhog on a mature repo usually surfaces dozens of pre-existing findings — many already rotated, allowlisted by other tools, or known examples. Baselines let you snapshot the current state and only fail CI on findings introduced after that snapshot.

Create the snapshot

$ keyhog scan . --create-baseline .keyhog-baseline.json

Runs a normal scan, then writes the findings to a baseline file. The baseline records each finding by (detector, sha256, path) — hash-keyed so renaming a file or moving a leak between lines does not silently drop it from the baseline.

Use the snapshot

$ keyhog scan . --baseline .keyhog-baseline.json

Scans normally, then filters the output: anything already in the baseline is suppressed; anything new is reported and exits 1. Run this in CI and the build only breaks when a developer commits a not-previously-known leak.

What "matches the baseline" means

A finding is considered baseline-matched when both hold:

  • Same (detector, sha256) pair (same secret value, same detector).
  • Path is unchanged or the new path is a rename of the baseline path (detected via git rename heuristic when scanning a git repo).

So rotating a leaked secret to a different (still-leaked) value will fire — same detector, new sha256. Moving a leaked secret to a different file without rotating will fire — same sha256, new path that isn't a rename.

Update the snapshot

$ keyhog scan . --update-baseline .keyhog-baseline.json

Adds new findings to the baseline while preserving every existing entry. Use this when you've reviewed today's new findings, confirmed they're either false positives or already-tracked work, and want to re-snapshot.

Never blindly run --update-baseline in CI. It hides real new findings under the assumption that "everything currently firing is fine." Use --baseline in CI; run --update-baseline manually after triage.

Workflow example

# Initial adoption
$ keyhog scan . --create-baseline .keyhog-baseline.json
$ git add .keyhog-baseline.json
$ git commit -m "adopt keyhog with baseline"

# In CI (.github/workflows/secrets.yml)
$ keyhog scan . --baseline .keyhog-baseline.json --format sarif -o keyhog.sarif

# After triaging a batch of new findings (some real, some accepted)
$ keyhog scan . --baseline .keyhog-baseline.json
3 new findings:
  src/test/fixtures.js:42  generic-api-key  [low · 0.45]  ← false positive, accepted
  prod/.env:7              stripe-secret    [critical]    ← REAL LEAK — rotate first!
  docs/example.md:11       aws-access-key   [high · 0.5]  ← example credential, accepted

# After rotating the real leak and accepting the two examples:
$ keyhog scan . --update-baseline .keyhog-baseline.json
$ git diff .keyhog-baseline.json   # review the snapshot delta
$ git commit .keyhog-baseline.json -m "baseline: accept fixture FPs"

diff: compare two scans without a stored baseline

$ keyhog scan . --format json -o before.json
# … land changes …
$ keyhog scan . --format json -o after.json
$ keyhog diff before.json after.json

Prints findings as NEW, RESOLVED, or UNCHANGED and exits 1 if any NEW rows landed. Use when you want diff-aware CI without committing a baseline file.

Diff-aware severity (git history)

When scanning git history (--git-history), keyhog downgrades findings one severity tier if the offending blob is not reachable from HEAD. A critical secret that was leaked in a historic commit but already rotated and removed from main ranks as high, not critical. Live leaks (still reachable from HEAD) keep their declared severity.

This rule sorts the noise: the things you can fix by force-pushing are ranked below the things an attacker can still find by reading main today.

Cross-detector baselines

A baseline file is detector-agnostic — adding new detectors to the embedded set won't invalidate existing baseline entries. New detectors that fire on existing content will surface as new findings on the next scan; once triaged, --update-baseline rolls them in.

Tip: commit .keyhog-baseline.json alongside .keyhogignore at the repo root. The baseline tracks "everything currently firing"; the ignore file tracks "individually blessed values with a written reason." Use the ignore file for individually-known leaks; use the baseline for adoption.
