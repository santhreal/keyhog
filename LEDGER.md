## BACKLOG: flate2 gzip/zip decode-through (RESEARCHED design — NOT a simple decoder add)
Root cause is deeper than "no inflate decoder": Chunk.data is a UTF-8 SensitiveString (crates/core/src/source.rs:30), and every text decoder (base64.rs) gates emission on String::from_utf8 — so binary container bytes never survive into the pipeline intact. The decode pipeline (decode/pipeline.rs, splice.rs) is TEXT-span oriented (with_extracted_value_spans / decode_candidate_refs_exact operate on extracted str spans). Correct fix options:
1. In Base64Decoder (and hex): BEFORE the from_utf8 gate, try inflate on the decoded BYTES (gzip 0x1f8b / zlib 0x78 / raw-deflate); if inflate yields UTF-8 text, emit that spliced sub-chunk. Recovers the common "secret gzip'd then base64'd" case with no architectural change — just a pre-gate branch in the byte path. flate2 is already a workspace dep (crates/sources/Cargo.toml) — add `flate2 = { workspace = true }` to crates/scanner/Cargo.toml.
2. For raw gzip/zip FILES (binary source): needs the binary source path to emit the inflated text as a chunk (sources side), since chunk.data can't carry the raw magic bytes.
3. zip: needs the `zip` crate (central-directory parse); STORED entries carry the plaintext but a CRC byte >=0x80 fails from_utf8 — same pre-gate-inflate/parse approach.
BOMB BOUND (Law 15): bound each inflate output (e.g. 16MB) under the existing pipeline cap MAX_DECODED_TOTAL_BYTES=64MB (decode/pipeline.rs:5); use flate2 streaming reader with .take(limit).
PIN TESTS to update when landed: gap/decoder_registry_default_composition.rs + regression_cpu_decoder_parity.rs (default_decoder_names list) + MAX_PROFILED_DECODERS headroom (16, currently 13). And flip the gzip/zip assertions in the committed regression_decode_through_recall.rs from gap-negative (count==0) to recovered-positive.
