# Dogfood findings (real tool)

Issues found by running the freshly-built release binary across its surfaces.
NOT detection accuracy (that is the SecretBench scorer's job).

## Confirmed robust (2026-05-30, v0.5.37 rebuild)
- All 7 output formats (`text json jsonl sarif csv junit html`) on a planted-secret dir: valid output, correct exit codes, NO panics. JSON/JSONL parse; SARIF has version+runs; HTML has DOCTYPE; CSV has header.
- Edge cases NO panic: nonexistent path (exit 2, correct), empty dir (exit 0), `--stdin`, 2MB binary file.
- Subcommands clean: `detectors` (44KB), `backend`, `doctor`, `completion bash` (41KB), `explain <valid-id>` (exit 0 with suggestions).
- **DoS robustness GOOD**: 20MB single line w/ no newline (exit 0, <20s), 12-deep base64 nesting (exit 0), symlink loop A↔B (exit 1, no hang). No timeouts, no panics, no OOM.
- baseline/diff/incremental round-trips: `--create-baseline` writes, `--baseline` rescan baselines correctly (exit 0), `--incremental-cache` written, `diff` identical→0. Exit-code contract 0=clean / 1=findings / 2=error holds for `scan`.

## Issues
- **DF-01 · DONE · detectors** — `detectors --format json` failed (exit 2); the flag was `--json`. FIXED with CLI-01: `--format <text|json>` is now the canonical flag, `--json` an alias. Verified by `tests/e2e/detectors_format_flag_matches_json_alias.rs`.
- **DF-02 · DONE · explain + FP_AUDIT** — `explain aws-access-key-id` → exit 2 "no detector with id". Closed with MC-10: the report no longer cites the drifted id, and `tests/gap/doc_detector_ids_resolve_in_registry.rs` guards every doc-cited detector id against the registry. (explain itself was always fine: helpful "Did you mean:" suggestions.)
- **DF-03 · DONE · sources/git (--git-staged outside a repo)** — running `scan --git-staged` outside a git repository leaked a raw git error. FIXED: `get_staged_files` (cli/src/sources.rs) now probes `git rev-parse --is-inside-work-tree` first and bails with a clean, actionable message ("'<path>' is not a git repository — `--git-staged` scans the git staging area… drop `--git-staged` to scan the working tree"). Guarded by `tests/gap/git_staged_non_repo_clean_error.rs` (non-zero exit, no clean summary, names the cause + flag, no raw `git diff failed` leak).
- **DF-04 · DONE · exit-code contract coherence** — The original note was partly stale/wrong. `--help` DOES carry an `EXIT CODES` block (`crates/cli/src/args.rs` `after_help`) and `docs/src/reference/exit-codes.md` documents 0/1/2/3/4/10/11/130. The live gap was a COHERENCE bug: both surfaces mislabeled "source-backend failure" and "detector-corpus audit failure" as exit `3`, but the code (`main.rs` `is_user_io_error` maps not-found/permission-denied → 2; `orchestrator/run.rs` `EXIT_SOURCE_FAILED = 2`; `diff`'s `ensure_baseline_input` bails with a string, no io error → 2) makes them exit `2`. Dogfood confirmed against the built binary: unknown flag → 2, `--git-history` on a non-repo → 2, `diff` with a missing baseline → 2 (the note claimed `3` for the latter two). FIXED `exit-codes.md` (table rows 2/3 + the `## 3` prose) and the `--help` `after_help` to match the code, and pinned the contract with binary-driven regressions `scan_unknown_flag_exits_2`, `scan_git_history_on_non_repo_exits_2`, `diff_missing_baseline_exits_2` (plus `scan_finds_planted_bedrock_key_and_returns_exit_1`) in `crates/cli/tests/e2e_binary.rs`. Exit `3` is genuinely rare now (only non-not-found/non-permission-denied I/O + GPU/hardware init).
- **DF-07 · DONE · detectors/discord-bot-token.toml + embedded-load fail-closed** — the detector was DEAD: discord-bot-token.toml failed to parse (`TOML parse error at line 34, column 35`) because the regex was a single-quoted TOML literal `'...'` whose char class embedded a `'`, so `''` ended the string — Discord bot tokens were UNDETECTABLE. FIXED two ways: (1) the regex is now a triple-quoted literal `'''(?i)discord…'''` (parses cleanly); (2) ROOT-CAUSE — the embedded-load path (`cli/src/orchestrator_config.rs::load_detectors_embedded_or_fail`) used to drop an unparseable embedded detector with a `tracing::debug!` then continue (a Law-10 SILENT fallback — exactly how a dead detector reached a benched release). It now FAILS CLOSED: any embedded TOML that doesn't parse aborts with an error naming every offender (the embedded set is compiled in, so a parse failure is always a build/source bug, never a user condition). Guarded at the source crate by `crates/core/tests/gap/every_embedded_detector_parses.rs` (parses the EXACT embedded slice the binary ships, not on-disk files — closes T-04/T-05; all 900 embedded detectors parse). → MC-16 closed.
