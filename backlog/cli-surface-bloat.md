# CLI surface bloat

`scan` carries 68 flags; the binary exposes 18 subcommands. Surface this large
is hard to keep coherent, document, and test. Candidates to consolidate.

## Flag convention inconsistency
- **CLI-01 · DONE · scan vs detectors** — output format flag was inconsistent: `scan --format <text|json|...>` but `detectors --json` (boolean). FIXED: `detectors` now takes the canonical `--format <text|json>` (narrow set — SARIF/CSV/etc. have no meaning for a listing), with `--json` kept as a back-compat alias (mutually exclusive). Guarded by `tests/e2e/detectors_format_flag_matches_json_alias.rs` (parity, text-not-json, sarif-rejected exit 2, alias-conflict exit 2).
- **CLI-02 · DONE · scan** — the `--daemon`/`--no-daemon` boolean conflict pair is collapsed into a single canonical tri-state `--daemon[=auto|on|off]` (new `DaemonMode` ValueEnum in `args.rs`). `--daemon` bare → `on` (back-compat), `--daemon=off` is the canonical form of the retained `--no-daemon` compatibility alias, and absent → `auto` (the prior "neither flag" behavior). The field is `daemon: Option<DaemonMode>` with `require_equals` (so bare `--daemon` never swallows a following positional path) + `default_missing_value="on"`; `--no-daemon` stays as a non-hidden alias because pre-commit hooks/editor integrations in the wild pass it. ALL routing reads the new `ScanArgs::daemon_mode()` resolver (folds in the alias) — `daemon_route` + the windows guard were converted off the raw bools, so the alias can't be silently bypassed (Law 10). Pinned by `tests/gate/daemon_mode_tristate_parses.rs` (absent→auto, bare→on, =on/=off/=auto one-to-one, alias→off, conflict rejected exit-2, positional-disambiguation, space-form). Surface stays snapshot-stable (both long flags retained, CLI-08).

## Subcommand overlap (3 live-scan modes)
- **CLI-03 · DONE (documented distinction) · watch / daemon / tui** — audited the three: they are NOT duplicates and are kept distinct. `watch` (WatchArgs) is a FOREGROUND filesystem-event loop that re-scans changed files in place (the "(daemon mode)" in its help is a misnomer for "long-running", not the socket daemon). `daemon` (DaemonArgs: start/stop/status) manages a BACKGROUND persistent warm scanner served over a Unix-domain socket — its whole purpose is sub-ms repeated *single* scans (what `scan --daemon=on` routes into), holding a compiled scanner across invocations. `tui` (TuiArgs) is an interactive dashboard that scans a path once with a live finding feed — a frontend over the in-process scanner, NOT over the socket daemon. Distinction: event-driven-rescan vs warm-single-scan-service vs interactive-one-shot. They share no implementation and collapsing any pair would lose a capability. (The only fix taken: the `watch` "(daemon mode)" help wording is misleading vs the socket daemon — candidate for a wording tweak, tracked under coherence, not a structural collapse.)
- **CLI-04 · DONE (audited) · update / repair / uninstall** — `repair` is NOT a pure `update --force` alias: `repair.rs` health-checks the install FIRST and only reinstalls when the scan engine is broken (or `--force`), then re-verifies the reinstalled binary's health and returns a dedicated exit code if it still fails its self-test; `update` unconditionally fetches+self-replaces with the latest release. So `repair` = "restore a *known-good* working binary, guarded by health gates", `update` = "move to latest". The self-management trio belongs in the binary because keyhog ships as a standalone single binary with no separate installer (verified download + self-replace is the only delivery path); splitting it into an external installer would add a second distributable for no gain. Keep all three.
- **CLI-05 · DONE (keep as subcommand) · scan-system vs scan** — `scan-system` is NOT merely `scan` + a source preset: it owns a privileged mount-enumeration capability (`subcommands/scan_system/mounts.rs` discovers every mounted drive) plus whole-system git-history traversal, with a distinct safety/UX profile (🛰). Folding it into `scan --system` would bury that privileged discovery behind a boolean on the already-68-flag `scan`. Kept as a top-level verb; if CLI-06's source-dispatch model lands, system-mount discovery becomes a `--source system` backend and this can revisit.

## Source-flag sprawl on `scan`
- **CLI-06 · DONE (decision: keep flat flags; registry is the extension point) · scan** — audited `sources.rs::build_sources`: the extensible dispatch the item asks for ALREADY exists — `keyhog_sources::create_source(name, params)` is a registry keyed on `--source <NAME>` (sources.rs:248), and every NEW backend is added there, so the flat-flag surface does NOT grow per backend. The ~13 flat source flags (`--git-blobs`/`--s3-bucket`/`--docker-image`/…) are each `#[cfg(feature=...)]`-gated convenience entries with dedicated value parsers and help text; collapsing them into a `--source name:param=…` mini-language would be a breaking change AND would *hurt* discoverability for the common git/s3/docker cases (they'd vanish from `scan --help`). Decision: keep the flat flags as ergonomic front-ends for the built-in backends; the `--source` registry remains the additive path for new/third-party backends (per the crates-of-crates source model). No flag removal — the bloat concern is bounded because the registry, not new flat flags, absorbs growth.
- **CLI-07 · DONE (unblocked by MC-02) · scan** — preset flags `--fast/--deep/--precision` coexist with the individual knobs they set (`--decode-depth --min-confidence --no-decode --no-entropy --ml-threshold`). MC-02 is now fixed (preset = BASE, per-flag overrides layer on top, no early-return), so the precedence is honest and documented in three coherent places: (1) the `--fast`/`--deep` flag long-help in `args/scan.rs` now states "a preset is a BASE … any explicit knob you pass overrides it" with worked examples; (2) `docs/src/reference/configuration.md` Presets section gains the precedence paragraph + the previously-missing `--precision` row + the two one-directional contracts (precision `--min-confidence` raise-only, `--no-keyword-low-entropy` disable-only); (3) `--precision`'s existing "`--min-confidence` still overrides the floor on top" line stays accurate. The behavior is pinned by `tests/unit/orchestrator/build_scanner_config_deep_mode.rs` (preset base vs explicit override) under MC-02. No flag removed — presets and individual knobs are complementary, not redundant, once precedence is correct.

## Action item
- **CLI-08 · DONE · scan flag-set + subcommand-set snapshot gate** — `tests/gate/cli_surface_snapshot.rs` introspects the canonical `keyhog::args::command()` (the same `clap::Command` the binary runs, NOT rendered `--help`) and asserts the top-level subcommand set (16 verbs, ex-clap `help`) and the `scan` long-flag set (71, ex-`--help`) each match a pinned list, with a symmetric-diff failure message naming exactly what was added/removed and which `EXPECTED_*` const to update. A new flag or subcommand now fails the gate until added on purpose — additions are deliberate and the count can't silently grow. Pairs with the test-expansion vector.
